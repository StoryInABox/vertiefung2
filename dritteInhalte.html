Was ist WebGL?

Die Qual der Wahl

WebGL (Web Graphics Library) ist eine Javascript-Programmierschnittstelle, mit deren Hilfe 3D-Inhalte beschleunigt im Browser dargestellt werden können. Kurz gesagt WebGl ist eine API (Application Programming Interface).

Was ist eine API?

Eine API ist Code, welcher es ermöglicht, Anwendungen miteinander zu kommunizieren. API‘s dienen dazu, verschiedene Plattformen und Datensätze miteinander zu verbinden und diese über eine einheitliche Sprache anzusprechen. Eine API ist somit eine Art Übersetzer. APIs gibt es schon so lange, wie es Computer gibt, allerdings haben sie seit dem Jahr 2000 an Bedeutung gewonnen, als Onlinehandelsplattformen wie Ebay und Amazon sie einsetzten, um ihre Websites zu optimieren.

Also WebGL fungiert als Übersetzer, damit dreidimensionale Inhalte problemlos auf jedem Gerät und Betriebssystem dargestellt werden können. Heutzutage ist WebGL in fast allen Web-Browsern standardmäßig enthalten. Kopf hinter WebGL ist die Non-Profit Vereinigung „Khronos-Gruppe“. Dieser Zusammenschluss aus über 150 branchenführenden Unternehmen im Bereich Grafik hat es sich zur Aufgabe gemacht, industrielle Standards bei der Darstellung von digitalen Inhalten zu schaffen. Dafür stehen sie im stetigen Austausch untereinander und mit weiteren großen Unternehmen wie Microsoft, Apple, Google, Nvidia, Intel etc. Die Khronos-Gruppe ist neben WebGL auch für andere APIs wie OpenGL (Open Graphics Library) oder das 3D-Format .gltf (Graphics Library Transmission Format, seit 2015).

Frameworks

Der große Alrounder

Man könnte jetzt also einfach WebGL lernen und damit seine 3D-Inhalte ins Internet bringen. Da WebGL aber sehr unhandlich ist, gibt es für dreidimensionale Anwendungen diverse Frameworks.

Was ist ein Framework?

Ein Framework ist kein eigenständiges Programm, sondern ein Hilfskonstrukt, welches einem die Arbeit mit einer bestimmten Software vereinfacht. Man spart sich also Code, Zeit und Nerven, wenn man ein Framework nutzt.

HTML Basics

Html

Bevor wir uns gleich dem Codeschreiben etwas genauer widmen, klären wir noch mal kurz die Basics im Bereich HTML. Denn egal welches Framework wir nutzen am Ende wird unsere Anwendung in Form einer HTML-Datei vorliegen.

Also nochmal zur Wiederholung: HTML, CSS, Javascript was war das?

Eine .html-Datei (Hypertext Markup Language) ist die Grundlage einer Website. Es ist eine reine Textdatei, in welcher die Elemente, die Struktur, die Funktionsweise und das Aussehen einer Website definiert ist. Oft gibt es zusätzlich noch .css-Dateien (Cascading Style Sheet) und .js-Dateien (Javascript), wobei man diese auch direkt in die .html-Datei integrieren kann. CSS benötigt man für das Aussehen und Javascript für die Funktionsweise einer Website.

Innerhalb eines HTML-Dokuments finden sich zwei Bereiche: der Head- und der Body-Teil.

Im Head-Teil werden grundlegende Dinge initialisiert: Titel der Seite, zu importierende Schriftarten und Skripte sowie das CSS-Dokument oder direkt der CSS-Code.

Im Body befinden sich die eigentlichen Inhalte der Website: Buttons, Bilder, Text usw.

Scripte können entweder in den Header oder den Body geschrieben werden, je nachdem, wo sie ausgeführt werden sollen.

A-frame

Der kleine Alrounder

Genug der Theorie. Kommen wir nun zu den Frameworks im Einzelnen. Fangen wir zunächst mit den codebasierten Frameworks an. Codebasiert heißt, wir schreiben unsere Anwendung in HTML, CSS und Javascript.

Starten wir mit A-Frame.

A-Frame wird seit 2017 von Don McCurdy entwickelt. A-Frame ist das Einfachste der codebasierten Frameworks. Da es prinzipiell erst mal kein JavaScript braucht. Wer nicht gerne Code schreibt, ist hier richtig.

A-Frame wurde mit einem Schwerpunkt auf VR- und XR-Anwendungen entwickelt. Wenn man sich die Beispielszene anschaut, sieht man, dass alles direkt im Body geschrieben wird und kein Script Teil benötigt wird. Dieser wird automatisch generiert.

Datei Import

Der große Vorteil von A-Frame ist mit Sicherheit die Größe. Mit gerade mal 12 Zeilen lässt sich ein eigenes 3D-Modell fürs Internet aufbereiten, mit der Möglichkeit, sich das Ganze auf dem Smartphone in VR anzuschauen. Das schafft kein anderes Framework!

Allerdings muss man aber dazu sagen, dass A-Frame kein eigenständiges Framework ist, da es auf Three.js basiert. Somit ist A-Frame ein Framework im Framework. Und hier ist auch der Hacken an der Sache. Es lassen sich zwar superschnell 3D-Modelle in Netz bringen, wenn man aber komplexe Projekte hat, kommt man früher oder später an den Punkt, dass man auf Three.js zurückgreifen muss. Siehe das Beispiel auf der nächsten Seite.

Datei Import

Da A-Frame z. B. standardmäßig keine Orbit-Controls bietet, müssen diese vorher aus Three.js importiert werden. Auch das beleuchten mit HDRI‘s benötigt zusätzliche Scripte. Alles Probleme, für die es Lösungen gibt. Allerdings mildern diese Einschränkungen und Weitere nicht genannte den Charme von A-Frame. Weshalb ich bisher immer direkt auf Three.js zurückgegriffen habe.

Three.js

Der kleine Alrounder

Wo wir gerade von Three.js sprechen. Three.js gibt es seit 2010 und wurde von Mr.doob entwickelt.

Unter den hier vorgestellten Frameworks ist Three.js der Community-Liebling. Zu keinem anderen Framework lassen sich mehr Einträge im Internet finden. Das erleichtert einem den Einstieg enorm, da es auf jede Frage online die passende Antwort gibt.

Was das Handling angeht, sieht es bei Three.js im Vergleich zu A-Frame etwas anders aus. Hier wird ausschließlich mit Javascript gecodet. Für diesen recht Simplen, sich drehenden grünen Würfel braucht es bereits über 50 Zeilen Code

Auch Hier könnt ihr die Szene selber anpassen

Datei Import

Das liegt an der Art, wie Three.js aufgebaut ist. Zu Beginn werden wie bei A-Frame im Head-Teil das Framework und die benötigten Komponenten geladen. Ähnlich wie beim HTML-Dokuments gibt es bei Three.js im Script zwei Teile bzw. zwei Grundfunktionen. Eine Init-Funktion und eine Render- oder Animate-Funktion.

Im Init-Teil werden alle Komponenten erstellt. Dazu gehören zwingend der WebGLRender, eine Kamera, eine Szene und idealerweise ein 3D-Objekt. Außer Kamera und Renderer werden alle Objekte der Szene hinzugefügt, sodass im Anschluss in der Render-Funktion der Renderer den Befehl erhält, Kamera und Szene zu rendern „renderer.render( scene, camera );“. Bei Animationen wir zusätzlich noch folgender Befehl genutzt „requestAnimationFrame( animate );“, durch welchen in diesem Fall die Animate-Funktion kontinuierlich ausgeführt wird.

Datei Import

Das klingt erst mal alles kompliziert, gilt aber für fast alle Anwendungen mit Three.js. Das ist allerdings auch der Knackpunkt an Three.js, da immer alles bis ins kleinste definiert werden muss wird bei komplexeren Szenen ein Script schnell ein paar Hundert Zeilen lang. Das Gute auf der anderen Seite ist, dass man alles wirklich alles ohne Einschränkungen umsetzten kann. Hier lohnt sich ein Blick auf die Beispiele und die umfangreiche Dokumentation. Man kann praktisch jedes 3D-Format importieren, hat diverse Postprocessing- Effekte, Raycasting, Physics, etc. sodass sich mit Three.js theoretisch auch Spiele umsetzten lassen.

Babylon.js

Der kleine Alrounder

Kommen wir zum letzten codebasierten Framework. Babylon.js!

Babylon.js gibt es seit 2013 und wurde von David Catuhe, einem Microsoft-Mitarbeiter, entwickelt.

Das Besondere an Babylon.js ist, dass man hier in zwei Programmiersprachen coden kann. Ursprünglich wurde Babylon.js für Typescript entwickelt und wurde dann für Javascript aufgearbeitet. Javascript hat unter Programmierern einen eher schlechten Ruf, da es nicht so strikten Regeln folgt wie andere Programiersprachen und daher etwas chaotisch bis willkürlich sein kann. Typescript ist eng mit Javascript verwandt und wurde von Microsoft genau dafür entwickelt, um die Mängel von Javascript zu kompensieren.

Ansonsten ähnelt Babylon.js, Three.js in vielen Punkten. Vom Code-Umfang der Beispielszene liegen wir hier auch bei ca. 50 Zeilen

Datei Import

Wie bei Three.js werden auch hier erst Renderer, Szene, Kamera etc. erstellt und dann anschließend gerendert. Der Unterschied zu Three.js ist allerdings die Funktionsweise von Init- und Render-Funktion und das alles insgesamt etwas kompakter und smarter angelegt ist. Zusätzlich verfügt Babylon.js über einen Online-Editor und den Node- Material-Editor (ähnlich dem Node-Material Editor bei Unity).

Datei Import

Dennoch fällt einem der Einstieg in Babylon.js etwas schwerer. Das mag zum einen an der gewöhnungsbedürftigen Dokumentation liegen, aber vor allem an der kleineren Community. Trotz allem ist Babylon auf lange Sicht das wohl das stabilere und ausgereiftere Framework. Da durch den verkürzten Code selbst komplexere Szenen seltener über die Hundert-Zeilen-Grenze kommen.

Fazit

Datei Import

Insgesamt muss man sagen, die Welt der WebGL Frameworks ist recht vielfältig und es sollte für jeden was dabei sein. Überrascht haben mich besonders die kleineren Vertreter wie A-Frame und Verge3D da sie doch einiges mehr können, als man zunächst vermutet.

Wenn ich jetzt eine Empfehlung für das beste Framework aussprechen müsste, würde ich wahrscheinlich Babylon.js sagen, da es im Bereich WebGL das stabilste und ausgereifteste Framework ist, welches zugleich noch sehr kompakt ist.

Allerdings korreliert die Wahl des Frameworks stark mit der Art der Anwendung. Somit lautet meine Empfehlung wie folgt.

  • Bei kleinen VR-/XR-Anwendungen: A-Frame.
  • Als Einsteiger: Verge3D, A-Frame oder Three.js (je nach Code-Affinität und Anwendung).
  • Bei sehr großen, komplexen Anwendungen: Unity.
